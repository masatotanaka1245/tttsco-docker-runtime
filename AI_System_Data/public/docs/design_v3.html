1要件定義書

1.1 システムの目的

社会インフラのライフサイクルを支える総合コンサルティング業務を、Windows本番サーバー上の Apache 2.4.57、PHP 8.2.8、MySQL 8.0.33、Ollamaを組み合わせたオンプレミス型Webアプリとして支援する。案件、資料、CSVデータ、コメント、FAQ、担当メンバーを一元管理し、PDF・CSVをRAG化してAIチャットから検索、要約、集計、分析できる業務支援基盤を構築する。

1.2 機能要件（資料・データ統合範囲の拡張）

本システムは、PDF等の報告書ドキュメントの意味解析に加え、ボーリングN値、気象観測値、材料試験結果といったCSV/TSV由来の構造化データの取り込み、表形式プレビュー、自然言語RAG化、Text-to-SQL分析を提供する。認証、CSRF、案件権限、project_membersによるアサイン制御を組み込み、案件単位で資料とナレッジを安全に扱う。

対象分野：

  • 河川、砂防及び海岸・海洋分野
  • 土質及び基礎分野
  • 港湾及び空港分野
  • 鋼構造及びコンクリート分野
  • 電力土木分野
  • トンネル分野
  • 道路分野
  • 建設環境分野
  • 都市計画及び地方計画分野
  • 電気電子分野
  • 地質分野

3.2 構造化データテーブル用 データベース定義 (DDL)

案件ごとにカラム構成が完全に異なるあらゆるCSVデータ群を高速に取り込み、一元的に管理するためのJSON動的格納スキーマ定義です。

MySQL Schema (DDL) InnoDB 
-- A. CSVデータテーブル インポートメタ管理
CREATE TABLE project_csv_files (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    project_id BIGINT UNSIGNED NOT NULL,
    file_name VARCHAR(255) NOT NULL,
    column_headers TEXT NOT NULL -- 例: ["測定日時", "地点", "N値"] のJSON配列,
    row_count INT NOT NULL DEFAULT 0,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (project_id) REFERENCES projects(id) ON DELETE CASCADE
) ENGINE=InnoDB;

-- B. CSVレコード（行） JSONデータ格納
CREATE TABLE project_csv_rows (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    csv_file_id BIGINT UNSIGNED NOT NULL,
    row_index INT NOT NULL,
    row_data JSON NOT NULL -- 例: {"測定日時": "2026/05/21", "N値": 32} 形式,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (csv_file_id) REFERENCES project_csv_files(id) ON DELETE CASCADE,
    INDEX idx_csv_file (csv_file_id, row_index)
) ENGINE=InnoDB;

4.2 案件管理・CRUD

案件は `projects` を中心に管理する。作成者は `created_by` に保存され、一般ユーザーは自分が作成した案件または `project_members` に登録された案件のみ参照できる。編集、削除、メンバー変更、PDF/CSV削除などの管理操作は `canManageProject()` により、admin、作成者、manager/member権限のメンバーに限定される。

主なAPI

  • `add_project.php`、`update_project.php`、`delete_project.php`
  • `add_comment.php`、`delete_comment.php`
  • `add_faq.php`、`delete_faq.php`
  • `add_project_member.php`、`remove_project_member.php`

フロントエンド制御

  • `support.js` がUI、チャット、CSV、地図、案件モジュールを統合
  • `modules/project.js` が実際のCRUD関数を `window` に公開
  • `modules/api.js` がCSRF付き `secureFetch()` を提供

4.3 PDF登録・RAG化

PDFアップロードは `upload.php` が担当する。画面から送られた `document` または互換用 `pdf` フィールドを受け取り、`public/01_RAG_Documents/{project_id}/` に保存する。`pdfinfo` でページ数を取得し、`pdftotext` でネイティブテキストを抽出する。画像化が必要な場合は `pdftoppm` でPNG化し、VLMへページ分類、2x2タイル、1x8水平スライス、全体解析を依頼する。抽出結果は500文字程度に分割され、Ollama Embedding APIでベクトル化したうえで `doc_chunks` に保存される。

4.4 CSV/TSV登録・外部PostgreSQL取込

`upload_csv.php` はCSV/TSVの文字コードと区切り文字を判定し、列名を `project_csv_files.column_headers` に、各行を `project_csv_rows.row_data` にJSONとして格納する。同時に `documents` に `[CSVデータ] ファイル名` の親レコードを作り、各行を「列名は値です」という自然文に変換して `doc_chunks` へ登録する。`import_postgresql.php` は外部PostgreSQLへ接続し、抽出結果をCSV取込と同じ構造に流し込む。

4.5 AIチャット・分析モード

`chat.php` はSSE形式のチャットエントリーポイントで、質問内容に応じて通常RAG、フル思考、データ分析、全社横断調査の各ルートへ分岐する。通常RAGは `chat_normal.php`、多段階推論は `chat_advanced.php`、Text-to-SQL集計は `chat_analysis.php`、admin専用の全社横断調査は `chat_global.php` が担当する。回答と質問は `chat_history` に保存され、推論過程は `chat_reasoning_steps`、品質評価は `chat_evaluations` に保存される。

4.6 ログ・運用設定

ログは `logs/` 配下に出力する。`upload_debug.log` はアップロードやCSV/PG取込の進捗、`chat_debug.log` はチャット関連の重要ログ、`app_error.log` はAPI内部エラーを記録する。本番では詳細例外を画面に出さず、チャット詳細ログは必要時のみ有効化する運用とする。

6.2 東設向けレイアウト設計方針

① 情報密度の最適化

一般的なB2Cの「余白重視」を排除。一画面に複数パネルを展開し、マトリクスビューやインライン編集を採用。技術者が視線移動を最小限に抑え、複雑な解析パラメータを瞬時に把握できる構成とします。

② RAG 検索とハイライト・相互データ同期

左側に業務関連の情報、右側に「対話UI」を配置。AIの回答内の参照箇所をクリックすると、PDF.jsが自動で該当ページへジャンプし、ハイライト表示する「思考の同期」を実装方針とする。 ★拡張: 参照元がCSVデータテーブルである場合は、自動的に「📊 CSVデータ」タブに画面が切り替わり、対象のCSVデータグリッドを非同期で自動プレビュー展開する連携を構築。

③ 「静かでシンプルな」ダッシュボード

視覚的ノイズを削減し、Delta（変化値）やバッジによる「異常値・未処理タスク」の提示に特化。水位異常や新着基準書など、次に取るべきアクションを優先的に表示。

④ モバイル・フィールド対応

現場での利用を考慮し、ドラッグ＆ドロップおよびカメラ撮影による即時アップロードをFAB（フローティングボタン）で提供。48px以上のタップターゲットとピンチズーム対応。

7AI回答生成 (RAG) アーキテクチャとシーケンス図

本システムにおけるAI対話パイプラインは、`public/api/chat.php` を入口としてSSEで進捗と最終回答を返す。ユーザー入力の内容、文字数、分析キーワード、全社横断キーワード、フロント側のモード指定に基づき、通常RAG、フル思考、Text-to-SQL分析、admin専用グローバル調査へ分岐する。

7.1 【軽量ルート】一問一答型RAG（文脈補正・一発検索）

`chat_normal.php` が担当する基本ルートです。質問ベクトルを生成し、案件内の `doc_chunks` を検索する。PDF本文、PDF画像説明、CSVから自然言語化された行データを同じ検索対象として扱い、関連チャンクをプロンプトに入れて回答を生成する。

7.2 【重厚ルート】フル思考モード

`chat_advanced.php` が担当する多段階推論ルートです。質問を複数観点へ分解し、観点ごとに資料・CSV・コメント・FAQを探索する。中間結果は `chat_reasoning_steps` に保存され、最終回答は統合モデルで生成する。進捗はSSEでリアルタイムにプッシュされます。

9データ分析エージェント (Text-to-SQL) ＆ グラフ自動生成

本システムは、RAGによる「文章の検索・要約」だけでなく、構造化データ（CSV等）に対する「正確な数値集計」と「視覚的なグラフ化」をチャット上で行うデータ分析エージェントを搭載しています。`SqlExecutionEngine.php` が生成SQLを監査し、SELECT系の安全なSQLだけを案件スコープ内で実行することで、LLM特有の数値ハルシネーションを抑制します。

9.1 JSON動的スキーマとText-to-SQLの統合

SQL Query Generation Context

任意の形式のCSVデータを project_csv_rows テーブルの row_data (JSON型) カラムに格納しています。AIには、MySQL 8.0 のJSON抽出関数（->>）を用いたクエリ生成を強制します。

// ユーザーの質問: 「土質区分ごとのN値の平均を計算して」
SELECT
  row_data->>'$.土質区分' AS soil_type,
  AVG(CAST(row_data->>'$.N値' AS DECIMAL(10,2))) AS avg_n_value
FROM project_csv_rows
WHERE csv_file_id = [対象のCSV ID]
GROUP BY row_data->>'$.土質区分';

10PDF/CSVファイルの処理・RAG化ロジック

非構造化ドキュメント（PDF）と構造化数値データ（CSV）という性質の異なるデータをどちらも等しくAIチャットの検索・精読対象にするため、それぞれ高度なパースロジックを搭載しています。

10.1 非構造化データ（PDF）のVLMマルチ解析 ＆ チャンク最適化

PDF資料は、ネイティブテキスト抽出とVLM（視覚言語モデル）のスマート・ルーティング判定により、ページごとに最適な方法で文字起こしを実行。細かな図面や横長の表組みに対しては、「2x2タイル分割」や「1x8水平スライス（8分割）」を施して高精度に文字情報を抽出します。

10.2 構造化データ（CSV）の自律自然言語化パイプライン

CSVのような数値がセルに並んだ構造化データは、意味的なつながり（文脈）を持たないため、AIチャットのセマンティック検索時に適合度が低くなります。これを解決するため、インポート時に「列名と行の値を機械的に結合して自然言語の文章に翻訳する」パイプラインを実装しています。

📝 変換プロセスの具体例:

【生CSVデータ（第12行目）】

測定日時: 2026/05/21 14:00
地点: A工区
深度(m): 15.2
N値: 35
土質区分: 砂礫

【RAG自動変換テキスト】

CSV「ボーリング柱状図.csv」の第12行のデータ：測定日時は「2026/05/21 14:00」、地点は「A工区」、深度(m)は「15.2」、N値は「35」、土質区分は「砂礫」です。

10.3 チャット ⇔ スプレッドシート プレビュー同期インタラクション

CSV翻訳テキストがヒットしてソースバッジとして表示された際、クリックすると get_csv_file_id.php API が稼働。画面中央のメインコンテナを一瞬で「📊 CSVデータ」タブへ切り替え、対象のグリッドビューを自動プレビュー展開させます。

10.4 進捗監視とキャンセル制御

PDF/CSV/外部PostgreSQLの重い処理では、セッションID単位の `logs/progress_{session_id}.json` に進捗を書き出し、`get_upload_status.php` がフロントエンドへ状態を返す。PDF解析は `cancel_{session_id}.flag` を検知して処理を中断できる。セッションロックを早期解放するため、アップロード中でもポーリングや別APIの応答がブロックされにくい。

10.5 CSV構造化データRAGの高速化・AI推論ボトルネック解消ロジック

同一ファイル由来の大量のCSV翻訳レコードがヒットした場合、精読ループ回数が肥大化し深刻な推論遅延ボトルネックとなっていました。この課題を解決するため、以下の機能をバックエンドAPIへ新規に設計・統合しました。

① 複数行チャンクの列同一性バルクアグリゲーション (Structured Bulk Compression)

ベクトル検索によって「同一のCSVファイル」に属する複数の翻訳行チャンクがヒットした場合、PHP側で自動的に「元の1つのMarkdownデータテーブル（表組み）として統合・再圧縮」を行います。

🔄 統合後のバルク圧縮テーブル (1リクエストに集約)
以下の表は、検索に合致した「ボーリング柱状図.csv」のレコード一覧です。 | 行番号 | 深度(m) | N値 | 土質区分 | | :--- | :--- | :--- | :--- | | 1 | 1.2 | 8 | 表土 | | 2 | 2.5 | 12 | 砂質土 |

これにより、AIの呼び出し回数が最大で 1/4 〜 1/10 に激減し、処理時間を70%以上短縮できます。

② CSV専用ダイナミック類似度トリアージ (Structured Dynamic Thresholding)

CSVデータは定型記述が繰り返されるため、類似度スコアが高く算出されやすい特性を持ちます。データ氾濫を防ぐため、システムは以下の動的閾値分岐を適用します。

  • PDF文書データ: 類似度スコア `0.35` 以上を網羅（標準設定）
  • CSV構造化データ: 類似度スコア閾値を **`0.50` 以上**へ自動引き上げ。データの完全性保証のため上限数は設けず全件対象とする。

10.6 大規模データロード時のコンテキスト防護（Context Guard）と空回答フォールバック

CSV検索において該当行が100行を超えるなど巨大なテーブルが生成された場合、AIサーバーの処理可能コンテキスト制限（Token Limit）やGPUメモリ限界を突破し、エラーなしの「空（0文字）」で応答を返す（サイレント・ドロップ）現象が発生します。これを防ぐための二重防護実装です。

① 文字数比例コンテキスト防護バッファ (Structured Context Guard)

類似度スコアの高い行から順番に結合し、生成されたMarkdownテーブルが累積で **`4,000文字`** に達した時点で、自動的にデータの追加を停止。テーブルの末尾にセーフガードによる省略注記を付加してAIへ渡します。

② 空レスポンス検知 ＆ 技術的フォールバック・リカバリ (Fallback Recovery)

万が一、AIからの推論回答が空（0文字）または通信タイムアウトした状態で終了した場合、API側で文字数を監査し、自動的に技術的フォールバックガイダンス応答を動的に組み立て、SSEで配信します。

11LLM-as-a-Judge による自律型評価と自己修正ループ

フル思考モードで生成された最終的な統合回答に対し、システムが裏側でAI自身（評価役）に品質を採点させます。スコアが閾値（合格点）に満たない場合は、AIが自らフィードバックを生成し書き直しを要求します。

11.1 評価観点（メトリクス）設計

1. Proactivity（提案力・先回り力）★最重要

事実を答えるだけでなく、専門家としての「ネクストアクション」「リスクの指摘」など、一歩踏み込んだ有益な示唆が含まれているか。

2. Faithfulness（事実への忠実性）

生成された回答が、インポートされた資料の内容のみに基づいているか。外部知識によるハルシネーション（幻覚）が混入していないか。

3. Answer Relevance（的確性・網羅性）

ユーザーが投げかけた質問の「すべての要素」に対して、真正面から過不足なく答えているか。論点がずれていないか。

4. Clarity（構造化・可読性）

専門用語が正しく使われているか。Markdownを用いて技術者が直感的に読みやすい構成になっているか。
