1要件定義書

1.1 システムの目的

社会インフラのライフサイクルを支える総合コンサルティング業務を、Windows本番サーバー上の Apache 2.4.57、PHP 8.2.8、MySQL 8.0.33、Ollamaを組み合わせたオンプレミス型Webアプリとして支援する。案件、資料、CSVデータ、コメント、FAQ、担当メンバーを一元管理し、PDF・CSVをRAG化してAIチャットから検索、要約、集計、分析できる業務支援基盤を構築する。

1.2 機能要件（資料・データ統合範囲の拡張）

本システムは、PDF等の報告書ドキュメントの意味解析に加え、ボーリングN値、気象観測値、材料試験結果といったCSV/TSV由来の構造化データの取り込み、表形式プレビュー、自然言語RAG化、Text-to-SQL分析を提供する。認証、CSRF、案件権限、project_membersによるアサイン制御を組み込み、案件単位で資料とナレッジを安全に扱う。

対象分野：

  • 河川、砂防及び海岸・海洋分野
  • 土質及び基礎分野
  • 港湾及び空港分野
  • 鋼構造及びコンクリート分野
  • 電力土木分野
  • トンネル分野
  • 道路分野
  • 建設環境分野
  • 都市計画及び地方計画分野
  • 電気電子分野
  • 地質分野

3.2 構造化データテーブル用 データベース定義 (DDL)

案件ごとにカラム構成が完全に異なるあらゆるCSVデータ群を高速に取り込み、一元的に管理するためのJSON動的格納スキーマ定義です。

MySQL Schema (DDL) InnoDB 
-- A. CSVデータテーブル インポートメタ管理
CREATE TABLE project_csv_files (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    project_id BIGINT UNSIGNED NOT NULL,
    file_name VARCHAR(255) NOT NULL,
    column_headers TEXT NOT NULL -- 例: ["測定日時", "地点", "N値"] のJSON配列,
    row_count INT NOT NULL DEFAULT 0,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (project_id) REFERENCES projects(id) ON DELETE CASCADE
) ENGINE=InnoDB;

-- B. CSVレコード（行） JSONデータ格納
CREATE TABLE project_csv_rows (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    csv_file_id BIGINT UNSIGNED NOT NULL,
    row_index INT NOT NULL,
    row_data JSON NOT NULL -- 例: {"測定日時": "2026/05/21", "N値": 32} 形式,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (csv_file_id) REFERENCES project_csv_files(id) ON DELETE CASCADE,
    INDEX idx_csv_file (csv_file_id, row_index)
) ENGINE=InnoDB;

4.2 案件管理・CRUD

案件は projects を中心に管理する。作成者は created_by に保存され、一般ユーザーは自分が作成した案件または project_members に登録された案件のみ参照できる。編集、削除、メンバー変更、PDF/CSV削除などの管理操作は canManageProject() により、admin、作成者、manager/member権限のメンバーに限定される。

主なAPI

  • add_project.php、update_project.php、delete_project.php
  • add_comment.php、delete_comment.php
  • add_faq.php、delete_faq.php
  • add_project_member.php、remove_project_member.php

フロントエンド制御

  • support.js がUI、チャット、CSV、地図、案件モジュールを統合
  • modules/project.js が実際のCRUD関数を window に公開
  • modules/api.js がCSRF付き secureFetch() を提供

4.3 PDF登録・RAG化

PDFアップロードは upload.php が担当する。画面から送られた document または互換用 pdf フィールドを受け取り、public/01_RAG_Documents/{project_id}/ に保存する。pdfinfo でページ数を取得し、pdftotext でネイティブテキストを抽出する。画像化が必要な場合は pdftoppm でPNG化し、VLMへページ分類、2x2タイル、1x8水平スライス、全体解析を依頼する。抽出結果は500文字程度に分割され、Ollama Embedding APIでベクトル化したうえで doc_chunks に保存される。

4.4 CSV/TSV登録・外部PostgreSQL取込

upload_csv.php はCSV/TSVの文字コードと区切り文字を判定し、列名を project_csv_files.column_headers に、各行を project_csv_rows.row_data にJSONとして格納する。現在はDB登録を先に完了し、その後でRAG用の自然言語チャンクとEmbeddingを作成する。大規模CSVでは複数行をまとめたRAGチャンクを生成し、Embeddingが連続失敗した場合もCSV本体の登録を維持する。import_postgresql.php は外部PostgreSQLへ接続し、抽出結果をCSV取込と同じ構造に流し込む。

4.5 AIチャット・分析モード

chat.php はSSE形式のチャットエントリーポイントで、まず質問を粗く因数分解し、通常RAG、advanced_hybrid.doc_extract、CSV証拠読解、CSV構造化集計（CSV-AGG）、CSV要約、admin専用全社横断調査の各ルートへ分岐する。ルーティング前に ChatRequestGuard.php が空入力、挨拶、誤送信に近い短文、報告書モードに不十分な指示を判定し、重い推論へ入る前に確認応答へ切り替える。通常RAGは chat_normal.php、多段階推論は chat_advanced.php、CSV証拠読解・CSV要約・構造化集計は chat_analysis.php、admin専用の全社横断調査は chat_global.php が担当する。回答と質問は chat_history に保存され、推論過程は chat_reasoning_steps、品質評価は chat_evaluations に保存される。高評価の回答は要点化され、FAQ候補として project_faqs に自動登録される。SupportController.php は保存済み推論ステップを再取得し、ブラウザ更新後もプロセス表示を復元する。

チャットUI描画

  • SSEの status、chunk、result を受信し、本文をリアルタイム描画
  • Markdown、Chart.js用 json:chart、Mermaid図をチャット内にJIT描画
  • support.phpの回答モードは「フル思考」「図解」「報告書」の小型スイッチで切替
  • 生成中ステータスは入力欄上の小さなバーに表示し、詳細は回答完了後の折りたたみで確認
  • Markedのコードブロック互換差異、DOMPurifyによる canvas と data-* 削除を考慮

CSV即答・証拠読解

  • 100行以下のCSV概要質問はOllamaを呼ばず、DBレコードから即時サマリー生成
  • 詳細読解が必要な場合は、先にDB検索で対象行を絞り込み、ヒット行だけをバッチ読解
  • 検索語が弱い大規模CSV質問は、全件AI読解ではなくメタデータと代表サンプルの概況回答へ切替
  • 日付集計質問は CSV-AGG に分岐し、日付列判定後の定番SQLで deterministic に件数集計する
  • 各段階の件数、検索語、ヒット件数、文字数、Ollama呼び出し時間を chat_debug.log に記録

図解・報告書モード

  • 図解モードは、回答理解に有効な場合のみChart.jsグラフまたはMermaid図を回答へ追加する。CSV日付集計では json:chart を deterministic に生成し、単一CSVは折れ線、複数CSVは棒グラフを基本とする。
  • 報告書モードは、チャット回答をそのままPDF化せず、結論 / 分析対象 / 根拠 / 留意点 / 推奨アクション / 出典の型へ最終整形してから、サーバー側でmPDFを優先してPDFファイルとして出力する。
  • 生成PDFは通常アップロードPDFと同じ保存領域に配置し、documentsへ親レコードとして登録する。
  • 報告書本文は生成時のテキストを直接チャンク化し、doc_chunksへ登録するため、再OCRせず高速に検索対象化できる。
  • 報告書モードでも入力が不明瞭な場合はPDF生成を行わず、確認応答で終了する。

4.6 ログ・運用設定

ログは logs/ 配下に出力する。upload_debug.log はアップロードやCSV/PG取込の進捗、chat_debug.log はチャットのルート判定、粗い質問因数分解、入力ガード、DB収集、SQL生成・監査、Ollama送受信、評価、履歴保存、最終回答本文までの詳細ログ、app_error.log はAPI内部エラーを記録する。support.phpにはadmin向けのリアルタイムログビューアがあり、chat_debug_tail.php で追記分をtail表示できる。[INPUT-GUARD]、[SMART-ROUTER]、[CSV-AGG]、[SQL-REPAIR-POLICY]、[EVAL-POLICY]、[FINAL-ANSWER] により、誤送信抑止、因数分解結果、構造化集計、SQL修復上限、品質評価上限、最終回答本文を確認できる。本番では詳細例外を画面に出さず、ログ肥大化に注意して運用する。

6.2 東設向けレイアウト設計方針

① 情報密度の最適化

一般的なB2Cの「余白重視」を排除。一画面に複数パネルを展開し、マトリクスビューやインライン編集を採用。技術者が視線移動を最小限に抑え、複雑な解析パラメータを瞬時に把握できる構成とします。

② RAG 検索とハイライト・相互データ同期

左側に業務関連の情報、右側に「対話UI」を配置。AIの回答内の参照箇所をクリックすると、PDF.jsが自動で該当ページへジャンプし、ハイライト表示する「思考の同期」を実装方針とする。 ★拡張: 参照元がCSVデータテーブルである場合は、自動的に「📊 CSVデータ」タブに画面が切り替わり、対象のCSVデータグリッドを非同期で自動プレビュー展開する連携を構築。

③ 「静かでシンプルな」ダッシュボード

視覚的ノイズを削減し、Delta（変化値）やバッジによる「異常値・未処理タスク」の提示に特化。水位異常や新着基準書など、次に取るべきアクションを優先的に表示。

④ モバイル・フィールド対応

現場での利用を考慮し、ドラッグ＆ドロップおよびカメラ撮影による即時アップロードをFAB（フローティングボタン）で提供。48px以上のタップターゲットとピンチズーム対応。

7AI回答生成 (RAG) アーキテクチャとシーケンス図

本システムにおけるAI対話パイプラインは、public/api/chat.php を入口としてSSEで進捗と最終回答を返す。ユーザー入力の内容、文字数、分析キーワード、CSV読解キーワード、全社横断キーワード、フロント側の回答モード指定に基づき、通常RAG、フル思考、Text-to-SQL分析、CSV検索先行読解、admin専用グローバル調査へ分岐する。入力が短すぎる、挨拶のみ、報告書生成に必要な意図が不足している場合は、ルーティング前に軽量な確認応答で終了する。

7.1 【軽量ルート】一問一答型RAG（文脈補正・一発検索）

9データ分析エージェント (Text-to-SQL) ＆ グラフ自動生成

本システムは、RAGによる「文章の検索・要約」だけでなく、構造化データ（CSV等）に対する「正確な数値集計」「検索先行の証拠読解」「視覚的なグラフ化」をチャット上で行うデータ分析エージェントを搭載しています。SqlExecutionEngine.php が生成SQLを監査し、SELECT系の安全なSQLだけを案件スコープ内で実行することで、LLM特有の数値ハルシネーションを抑制します。小規模CSVの概要質問では、AI推論を待たずにDBレコードから即時サマリーを生成します。

9.1 JSON動的スキーマとText-to-SQLの統合

SQL Query Generation Context

任意の形式のCSVデータを project_csv_rows テーブルの row_data (JSON型) カラムに格納しています。AIには、MySQL 8.0 の JSON_UNQUOTE(JSON_EXTRACT(row_data, '$."キー"')) 形式を基本とする安全なJSON抽出を指示し、日本語キーや記号入りキーでも構文エラーを避けるよう制約します。

// ユーザーの質問: 「土質区分ごとのN値の平均を計算して」
SELECT
  row_data->>'$.土質区分' AS soil_type,
  AVG(CAST(row_data->>'$.N値' AS DECIMAL(10,2))) AS avg_n_value
FROM project_csv_rows
WHERE csv_file_id = [対象のCSV ID]
GROUP BY row_data->>'$.土質区分';

不正構文、所有権不一致、SELECT以外、FROM句のないダミーSQLを拒否し、失敗SQLとMySQLエラーを修復プロンプトへ渡す。

10PDF/CSVファイルの処理・RAG化ロジック

非構造化ドキュメント（PDF）と構造化数値データ（CSV）という性質の異なるデータをどちらも等しくAIチャットの検索・精読対象にするため、それぞれ高度なパースロジックを搭載しています。

10.1 非構造化データ（PDF）のVLMマルチ解析 ＆ チャンク最適化

PDF資料は、ネイティブテキスト抽出とVLM（視覚言語モデル）のスマート・ルーティング判定により、ページごとに最適な方法で文字起こしを実行。細かな図面や横長の表組みに対しては、「2x2タイル分割」や「1x8水平スライス（8分割）」を施して高精度に文字情報を抽出します。

10.2 構造化データ（CSV）の自律自然言語化パイプライン

CSVのような数値がセルに並んだ構造化データは、意味的なつながり（文脈）を持たないため、AIチャットのセマンティック検索時に適合度が低くなります。これを解決するため、インポート時に「列名と行の値を機械的に結合して自然言語の文章に翻訳する」パイプラインを実装しています。

📝 変換プロセスの具体例:

【生CSVデータ（第12行目）】

測定日時: 2026/05/21 14:00
地点: A工区
深度(m): 15.2
N値: 35
土質区分: 砂礫

【RAG自動変換テキスト】

CSV「ボーリング柱状図.csv」の第12行のデータ：測定日時は「2026/05/21 14:00」、地点は「A工区」、深度(m)は「15.2」、N値は「35」、土質区分は「砂礫」です。

10.3 チャット ⇔ スプレッドシート プレビュー同期インタラクション

CSV翻訳テキストがヒットしてソースバッジとして表示された際、クリックすると get_csv_file_id.php API が稼働。画面中央のメインコンテナを一瞬で「📊 CSVデータ」タブへ切り替え、対象のグリッドビューを自動プレビュー展開させます。

10.4 進捗監視とキャンセル制御

PDF/CSV/外部PostgreSQLの重い処理では、セッションID単位の logs/progress_{session_id}.json に進捗を書き出し、get_upload_status.php がフロントエンドへ状態を返す。PDF解析は cancel_{session_id}.flag を検知して処理を中断できる。セッションロックを早期解放するため、アップロード中でもポーリングや別APIの応答がブロックされにくい。

10.5 CSV構造化データRAGの高速化・AI推論ボトルネック解消ロジック

同一ファイル由来の大量のCSV翻訳レコードがヒットした場合、精読ループ回数が肥大化し深刻な推論遅延ボトルネックとなっていました。この課題を解決するため、質問意図に応じて「DB即答」「DB検索先行の証拠読解」「Text-to-SQL集計」「大規模CSV概況回答」を切り替えるルーティングを実装しています。

① 小規模CSV即答ルート (CSV-SUMMARY)

「CSVの内容をまとめて」「どんなデータが入っているか」などの概要質問で、対象CSVが100行以下の場合はOllamaを呼ばず、project_csv_files と project_csv_rows からファイル数、行数、列名、値サンプル、内容の見立てをPHP側で生成します。これにより10件程度のデータでは数分のAI待ちを避け、DB取得と履歴保存だけで回答できます。

② 複数行チャンクの列同一性バルクアグリゲーション (Structured Bulk Compression)

ベクトル検索によって「同一のCSVファイル」に属する複数の翻訳行チャンクがヒットした場合、PHP側で自動的に「元の1つのMarkdownデータテーブル（表組み）として統合・再圧縮」を行います。

🔄 統合後のバルク圧縮テーブル (1リクエストに集約)
以下の表は、検索に合致した「ボーリング柱状図.csv」のレコード一覧です。 | 行番号 | 深度(m) | N値 | 土質区分 | | :--- | :--- | :--- | :--- | | 1 | 1.2 | 8 | 表土 | | 2 | 2.5 | 12 | 砂質土 |

これにより、AIの呼び出し回数が最大で 1/4 〜 1/10 に激減し、処理時間を70%以上短縮できます。

③ CSV検索先行トリアージ (CSV-SEARCH)

CSVデータは定型記述が繰り返されるため、類似度スコアだけでは対象行が膨らみやすい特性を持ちます。現在は、質問文から検索語を抽出し、MySQLで先に row_data、ファイル名、列ヘッダーを検索してから読解対象を決めます。

  • 検索語あり: 該当行を最大300件まで読み込み、ヒット件数と制限有無をログに残す。
  • 検索ヒットなし: 全件AI読解へ進まず、検索語と登録CSV範囲を回答して再質問しやすくする。
  • 検索語が弱い大規模CSV: メタデータと代表サンプルから概況回答を返し、処理時間の肥大化を避ける。

④ CSV証拠読解ルートの詳細トレース

AI読解が必要な場合でも、検索語抽出、検索ヒット件数、DBレコード収集、バッチ分割、バッチAI読解、統合AI回答、履歴保存の各段階で、対象行数、文字数、レスポンス文字数、経過秒を chat_debug.log に出力します。遅延がDB検索なのかOllama推論なのかをログから切り分けられます。

10.6 大規模データロード時のコンテキスト防護（Context Guard）と空回答フォールバック

CSV検索において該当行が100行を超えるなど巨大なテーブルが生成された場合、AIサーバーの処理可能コンテキスト制限（Token Limit）やGPUメモリ限界を突破し、エラーなしの「空（0文字）」で応答を返す（サイレント・ドロップ）現象が発生します。これを防ぐための二重防護実装です。

① 文字数比例コンテキスト防護バッファ (Structured Context Guard)

類似度スコアやDB検索で抽出された行を順番に結合し、生成されたMarkdownテーブルが累積で 4,000文字 程度に達した時点で、自動的にデータの追加を停止。テーブルの末尾にセーフガードによる省略注記を付加してAIへ渡します。

② 空レスポンス検知 ＆ 技術的フォールバック・リカバリ (Fallback Recovery)

万が一、AIからの推論回答が空（0文字）または通信タイムアウトした状態で終了した場合、API側で文字数を監査し、自動的に技術的フォールバックガイダンス応答を動的に組み立て、SSEで配信します。

11LLM-as-a-Judge による自律型評価と自己修正ループ

フル思考モードや分析ルートで生成された最終回答に対し、システムが裏側でAI自身（評価役）に品質を採点させます。通常RAGでは ChatEvaluationPolicy.php が、根拠要求・長文・複数資料参照・図解/報告書モードなど評価が必要なケースだけを選別します。スコアが閾値（合格点）に満たない場合は、モード別上限の範囲でフィードバックを使って書き直しまたは追加抽出を行います。基準値を大きく上回る回答は、質問と回答を要点化したうえでFAQへ自動登録します。

11.1 評価観点（メトリクス）設計

1. Proactivity（提案力・先回り力）★最重要

事実を答えるだけでなく、専門家としての「ネクストアクション」「リスクの指摘」など、一歩踏み込んだ有益な示唆が含まれているか。

2. Faithfulness（事実への忠実性）

生成された回答が、インポートされた資料の内容のみに基づいているか。外部知識によるハルシネーション（幻覚）が混入していないか。

3. Answer Relevance（的確性・網羅性）

ユーザーが投げかけた質問の「すべての要素」に対して、真正面から過不足なく答えているか。論点がずれていないか。

4. Clarity（構造化・可読性）

専門用語が正しく使われているか。Markdownを用いて技術者が直感的に読みやすい構成になっているか。

11.3 高評価回答のFAQ自動登録

FaqAutoRegistrar.php は、回答評価が十分に高く、差し戻し対象ではなく、タイムアウトやエラー応答でもない場合に、project_faqs へ自動登録します。登録時はチャット全文をそのまま保存せず、質問は短い要約に、回答はコードブロックや図表定義を除いた要点に整形します。保存条件を満たさなかった場合も、[FAQ-AUTO] ログで理由を追跡できます。

  • 重複登録を避けるため、元の chat_history_id に加え、正規化した質問要約と回答要約を確認する。
  • 評価プロセスのフェイルセーフや初期ドラフト採用時は自動登録しない。
  • 通常チャット、分析チャット、フル思考チャットの履歴保存トランザクション内で実行する。

11.4 評価・修復の待ち時間抑制

通常チャットまで常に重い評価・修復ループへ入ると、回答表示後から保存完了までの待ち時間が長くなる。現行実装では、評価の必要性を先に判定し、SQL自己修復と評価差し戻しの上限をモード別に制御する。

  • 通常RAGの短い回答や根拠不要の軽量回答は、品質評価をスキップする。
  • SQL自己修復は通常1回、図解・報告書モードは2回を上限の目安とする。
  • 品質評価の差し戻しも通常1回、図解・報告書モードは2回を上限の目安とする。
  • 上限値は [SQL-REPAIR-POLICY] と [EVAL-POLICY] として chat_debug.log に記録する。
