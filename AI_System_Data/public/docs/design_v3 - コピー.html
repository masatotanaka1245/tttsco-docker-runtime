1要件定義書

1.1 システムの目的

社会インフラのライフサイクルを支える総合コンサルティング業務を、PHP 8.2.8 と MySQL 8.0.33 を活用して高度化・効率化する。原子力施設や国家重要インフラに関する機微データを扱うことから、外部ネットワークから論理的・物理的に隔離された完全オンプレミス環境において、IT/AIで連携した、コンサルティング業務の基盤を構築する。

1.2 機能要件（資料・データ統合範囲の拡張）

本システムは、PDF等の報告書ドキュメントの意味解析（Standard RAG）に加え、ボーリングN値、気象観測値、材料試験結果といったExcel/CSVから抽出される「構造化データテーブル」の高速取り込み、スプレッドシート表示、およびAIチャット（RAG）での定量検索・自動分析機能を完全オンプレミス環境下で一元提供します。

対象分野：

  • 河川、砂防及び海岸・海洋分野
  • 土質及び基礎分野
  • 港湾及び空港分野
  • 鋼構造及びコンクリート分野
  • 電力土木分野
  • トンネル分野
  • 道路分野
  • 建設環境分野
  • 都市計画及び地方計画分野
  • 電気電子分野
  • 地質分野

3.2 構造化データテーブル用 データベース定義 (DDL)

案件ごとにカラム構成が完全に異なるあらゆるCSVデータ群を高速に取り込み、一元的に管理するためのJSON動的格納スキーマ定義です。 

-- A. CSVデータテーブル インポートメタ管理
CREATE TABLE project_csv_files (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    project_id BIGINT UNSIGNED NOT NULL,
    file_name VARCHAR(255) NOT NULL,
    column_headers TEXT NOT NULL -- 例: ["測定日時", "地点", "N値"] のJSON配列,
    row_count INT NOT NULL DEFAULT 0,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (project_id) REFERENCES projects(id) ON DELETE CASCADE
) ENGINE=InnoDB;

-- B. CSVレコード（行） JSONデータ格納
CREATE TABLE project_csv_rows (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    csv_file_id BIGINT UNSIGNED NOT NULL,
    row_index INT NOT NULL,
    row_data JSON NOT NULL -- 例: {"測定日時": "2026/05/21", "N値": 32} 形式,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (csv_file_id) REFERENCES project_csv_files(id) ON DELETE CASCADE,
    INDEX idx_csv_file (csv_file_id, row_index)
) ENGINE=InnoDB;

6.2 東設向けレイアウト設計方針

① 情報密度の最適化

一般的なB2Cの「余白重視」を排除。一画面に複数パネルを展開し、マトリクスビューやインライン編集を採用。技術者が視線移動を最小限に抑え、複雑な解析パラメータを瞬時に把握できる構成とします。

② RAG 検索とハイライト・相互データ同期

左側に業務関連の情報、右側に「対話UI」を配置。AIの回答内の参照箇所をクリックすると、PDF.jsが自動で該当ページへジャンプし、ハイライト表示する「思考の同期」を実装方針とする。 ★拡張: 参照元がCSVデータテーブル（構造化データ）である場合は、自動的に「📊 CSVデータ」タブに画面が切り替わり、対象のCSVデータグリッドを非同期で自動プレビュー展開する、相互データ同期インタラクションを構築。

③ 「静かでシンプルな」ダッシュボード

視覚的ノイズを削減し、Delta（変化値）やバッジによる「異常値・未処理タスク」の提示に特化。水位異常や新着基準書など、次に取るべきアクションを優先的に表示。

④ モバイル・フィールド対応

現場での利用を考慮し、ドラッグ＆ドロップおよびカメラ撮影による即時アップロードをFAB（フローティングボタン）で提供。48px以上のタップターゲットとピンチズーム対応。

7AI回答生成 (RAG) アーキテクチャとシーケンス図 (通常・フル思考・CSV統合検索)

本システムにおけるAI対話パイプラインは、ユーザーへのUI上にはモード切り替えスイッチを持たせず、ユーザーが入力した質問の「文字数」や「複雑性を示すキーワード（比較、検証、留意点など）」に基づいて、システム内部で自律的に「軽量ルート」と「重厚ルート」を動的に切り替える（スマート・ルーティング）設計としています。
これにより、ユーザーはモードを一切意識することなく、シンプルな質問には数秒で応答し、複雑な検証には高度なマルチエージェント推論を自動実行させることができます。

7.1 【軽量ルート】一問一答型RAG（文脈補正・一発検索）

システムが「質問が50文字未満であり、かつ複雑な推論を要しない」と自動判定した場合に稼働する軽量な検索ルートです。
推論負荷を抑え、数秒で迅速に事実を提示することに特化しています。

7.2 【重厚ルート】フル思考モード（多段階網羅推論・マルチエージェント構成）

システムが「比較」「検証」などのキーワードや50文字以上の長文を検知すると、自律的にこちらのフル思考モードへ移行します。
このルートでは適材適所のマルチエージェント構成を採用し、深い推論とデータ抽出（頭脳役）には `gemma4:e4b` を、最終的な日本語の統合・レポート作成（書記役）には `gpt-oss:20b` を使い分けることで、ハルシネーションと他言語への脱線を完全に防止します。セッションのロックを即時解放し、進捗はSSE（Server-Sent Events）でリアルタイムに画面へプッシュされます。

9データ分析エージェント (Text-to-SQL) ＆ グラフ自動生成

本システムは、RAGによる「文章の検索・要約」だけでなく、構造化データ（CSV等）に対する「正確な数値集計」と「視覚的なグラフ化」をチャット上で行うデータ分析エージェントを搭載しています。
AI自身にSQLを記述・実行させることで、LLM特有の「数値のハルシネーション（計算間違い）」を完全に防ぎ、100%正確な集計レポートを提供します。

9.1 JSON動的スキーマとText-to-SQLの統合

本システムでは、任意の形式のCSVデータを project_csv_rows テーブルの row_data (JSON型) カラムに格納しています。
AIには、MySQL 8.0 のJSON抽出関数（->>）を用いたクエリ生成を強制するシステムプロンプトを組み込むことで、動的なスキーマに対しても柔軟にSQLを生成させます。

// ユーザーの質問: 「土質区分ごとのN値の平均を計算して」
// AIが自動生成するSQLクエリの例:
SELECT
  row_data->>'$.土質区分' AS soil_type,
  AVG(CAST(row_data->>'$.N値' AS DECIMAL(10,2))) AS avg_n_value
FROM project_csv_rows
WHERE csv_file_id = [対象のCSV ID]
GROUP BY row_data->>'$.土質区分';

10PDF/CSVファイルの処理・RAG化ロジック

本システムでは、非構造化ドキュメント（PDF）と構造化数値データ（CSV）という性質の異なるデータをどちらも等しくAIチャットの検索・精読対象（ハイブリッドRAG）にするため、それぞれ以下のような高度なパースロジックを搭載しています。

10.1 非構造化データ（PDF）のVLMマルチ解析 ＆ チャンク最適化

PDF資料（設計基準、地質報告書など）は、ネイティブテキスト抽出とVLM（視覚言語モデル）のスマート・ルーティング判定により、ページごとに最適な方法で文字起こしを実行。細かな図面や横長の表組みに対しては、「2x2タイル分割」や「1x8水平スライス（8分割）」を施して高精度に文字情報を抽出します。

さらに、抽出された重複テキストをLLMで一度マージ・クレンジングしたのち、検索精度が最も安定する500文字前後の「チャンク（塊）」に自動分割してベクトル（`embedding`）化を施し、`doc_chunks` テーブルに保存します。

10.2 構造化データ（CSV）の自律自然言語化（セマンティック翻訳）パイプライン

CSVのような「数値がセルに並んだ構造化データ」は、そのままの状態でベクトルデータベースに投入しても、意味的なつながり（文脈）を持たないため、AIチャットのセマンティック検索時に引っかからないか、あるいは極めて低い適合度でしかヒットしません。

これを完全に解決するため、`upload_csv.php` へのインポート時に、「列名（ヘッダー）と行の値（フィールドデータ）を機械的にガッチャンコして自然言語の文章に翻訳する」という「セマンティック翻訳」パイプラインを実装しています。

📝 変換プロセスの具体例:

【生CSVデータ（第12行目）】

測定日時: 2026/05/21 14:00
地点: A工区
深度(m): 15.2
N値: 35
土質区分: 砂礫

【RAG自動変換テキスト（チャンクテキスト）】

CSV「ボーリング柱状図.csv」の第12行のデータ：測定日時は「2026/05/21 14:00」、地点は「A工区」、深度(m)は「15.2」、N値は「35」、土質区分は「砂礫」です。

この自動生成された自然言語テキスト（チャンク）に対して `EmbeddingEngine` を通して高次元のベクトル値を算出し、PDFのテキストと同列のデータとして `doc_chunks` に保存。さらに `image_description` カラムにメタキーワードとして `CSVデータ行レコード` を強制付与して格納します。

💡 効果: これにより、AIチャット画面で「A工区の深度15mのN値はどうなってる？」と尋ねた際、セマンティック検索エンジンがこの高次元ベクトルの一致を瞬時に検知。PDF報告書の文脈とCSVの数値を完全に横断マージし、「P.12の耐震基本方針と、CSVのN値35を統合すると、〜工法が最適です」といった極めて専門的で精密な定量回答を出力することが可能になります。

10.3 チャット ⇔ スプレッドシート プレビュー同期インタラクション

AIのセマンティック検索によりCSV翻訳テキストがヒットして回答内に根拠（ソースバッジ）として表示された際、それをユーザーがクリックすると、背後にある get_csv_file_id.php API が稼働。 対象ドキュメントに紐づく物理CSVファイルのID（csv_file_id）を動的逆算し、画面中央のメインコンテナを一瞬で「📊 CSVデータ」タブへ自律スイッチしたのち、対象の構造化スプレッドシート（グリッドビュー）を自動プレビュー展開させます。これにより、PDFの自動ページスクロールに並ぶ、直感的でシームレスなUI/UX連携を実現しています。

10.4 CSV構造化データRAGの超高速化・AI推論ボトルネック解消ロジック

多段階推論（フル思考モード）では、検索でヒットした `doc_chunks` データを「最大5件ずつの精読グループ」に小分け（`array_chunk`）し、グループごとに中間検証をループ実行します。 PDFのような非構造化ドキュメントではこれで問題ありませんが、同一ファイル由来の大量のCSV翻訳レコード（行データ）がヒットした場合、この精読ループ回数が極端に肥大化（例: 20行ヒット ➡️ 4回ループ ➡️ Ollamaの処理時間が4倍以上に膨張）し、レスポンス速度を著しく損ねる深刻な推論遅延ボトルネックとなっていました。

この課題をシステム工学的に解決するため、バックエンドの制御構造へ以下の**「自律データバルクアグリゲーション（Structured Bulk Compression）」** および **「CSV専用ダイナミック・トリアージ（RAG-Threshold Tuning）」** を新規に設計・統合しました。

① 複数行チャンクの列同一性バルクアグリゲーション (Structured Bulk Compression)

ベクトル検索によって「同一のCSVファイル」に属する複数の翻訳行チャンクがヒットした場合、それらを1行ずつ小分けにして何度もLLM（Ollama）を叩くのではなく、PHP側で自動的に**「元の1つのMarkdownデータテーブル（表組み）として統合・再圧縮」**を行います。

// 従来の5件精読コンテキスト（5回別々に推論を実行） 「CSV第1行：深度1.2m, N値8」「CSV第2行：深度2.5m, N値12」「CSV第3行：深度3.8m, N値15」... // 🔄 統合後のバルク圧縮テーブル（これらを1回の推論、1リクエストに統合！） 以下の表は、検索に合致した「ボーリング柱状図.csv」のレコード一覧です。 | 行番号 | 深度(m) | N値 | 土質区分 | | :--- | :--- | :--- | :--- | | 1 | 1.2 | 8 | 表土 | | 2 | 2.5 | 12 | 砂質土 | | 3 | 3.8 | 15 | 粘性土 |

これにより、AIは「表組みデータ」として一度にコンテキストを把握できるため、推論精度が格段に向上（コンテキスト把握能力の強化）するだけでなく、AIの呼び出し回数が**最大で 1/4 〜 1/10 に激減し、処理時間を70%以上短縮**できます。

② CSV専用ダイナミック類似度トリアージ (Structured Dynamic Thresholding)

CSVから自然言語変換されたチャンク（例: `image_description = 'CSVデータ行レコード'` のもの）は、定型記述が繰り返されるため、通常の文章（PDFのテキスト）に比べてベクトル類似度スコアが極めて高く算出されやすい特性を持ちます。 そのため、一律のスコア閾値（`0.35` 以上）を適用すると、類似データが一度に数十〜数百行ヒットしてしまいます。

このデータ氾濫を防ぐため、検索クエリを発行する際にシステムは以下の**動的閾値分岐**を適用します。

  • PDF文書データ: 類似度スコア `0.35` 以上を網羅（標準設定）
  • CSV構造化データ: 類似度スコア閾値を **`0.50` 以上**へダイナミック自動引き上げ。★ハードリミット（上位15行等）は設けず、コサイン類似度スコア0.50以上の該当データを上限なくすべて網羅的に対象とする。これによりデータの完全性と連続性を100%保証する。

このダイナミック・フィルタリングにより、ノイズとなる類似行が推論プロセスに紛れ込むのを100%遮断し、本当にユーザーの質問に必要な「特異的な数値が記載された対象行」だけをピンポイントで超高速精読・比較分析します。

10.5 大規模データロード時のコンテキスト防護（Context Guard）と空回答フォールバック

CSV検索において上限を撤廃して全行を対象にした場合、該当行が100行を超えるなど想定以上の巨大なMarkdownテーブルが生成される可能性があります。 これをそのままローカルAIサーバー（Ollama / vLLM）に入力すると、AIが自身の処理可能コンテキスト制限（Token Limit）やGPUメモリ限界を突破し、エラーを返すことなく「空（0文字）」で応答を返す（サイレント・ドロップ）現象が発生します。

この致命的な現象を防ぎ、システムとしての信頼性と耐障害性を100%確立するために、以下の**二重安全防護ガードレール**をバックエンドAPIに統合実装しています。

① 文字数比例コンテキスト防護バッファ (Structured Context Guard)

データの一意性と完全性を尊重しつつも、Ollamaのコンテキスト限界（`num_ctx: 8192〜16384`、日本語でおよそ8,000文字〜1.5万文字強に相当）を超えないように、Markdownテーブル作成処理の中で**「累積文字数による安全切り詰め（コンテキスト防護）」**を行います。

類似度スコアの高い行から順番に結合していき、生成されたMarkdownテーブルが累積で **`4,000文字`（オンプレミス環境のGPU推論能力に最適化された安全制限閾値）** に達した時点で、自動的にデータの追加を停止。テーブルの末尾に以下の注記を付加してAIへ渡します。

「...（※他、類似スコアの低い [N] 件のデータ行レコードは、AIサーバーの処理限界を保護するためのコンテキストセーフガードにより省略されました。必要に応じて、質問内容をより具体的に指定して絞り込んでください）」

これにより、AIサーバーはパニックを起こすことなく、入手できた「最高類似度の高精度な中核データ」に集中して100%安定してストリーム推論を実行できるようになります。

② 空レスポンス検知 ＆ 技術的フォールバック・リカバリ (Fallback Recovery)

万が一、通信の極小の揺れやAIサーバー側の高負荷、メモリフラグメンテーション等に起因して、AIからの推論回答が空（0文字）または通信タイムアウトした状態で終了した場合、空のチャット履歴（ゴミデータ）を保存するのを防止します。

API側でAIの出力文字数を監査し、もし `0文字` または応答なしを検知した場合は、自動的に以下の**フォールバック・技術ガイダンス応答**を動的に組み立て、データベースに正常記録し、かつ画面上にプッシュ配信（SSE）します。

「大変申し訳ありません。検索に合致したデータ量が多すぎるか、AIサーバーが一時的に極めて高負荷なため、処理能力限界（Token Limit）を超過いたしました。お手数ですが、質問に『特定の工区名』『特定の測定日時』、または『深度15m付近』などの詳細な絞り込みキーワードを添えて、再度お試しください。」

11システム名称候補

現在仮称となっている「TEPSCO Routines」以外の、本システムのコンセプト（AIによる高度な解析、全社のナレッジ統合、未来の働き方）をより強調する代替ネーミング案です。今後のブランディング検討の材料として記録します。

1. 知見・ナレッジの統合を強調

  • TEPSCO Brain (テプスコ・ブレイン)
    全11部門の資料やノウハウが蓄積され、AIが思考する「会社の頭脳」であることを表現。
  • TEPSCO Insight (テプスコ・インサイト)
    単なるデータ検索ではなく、図面や報告書から深い「洞察（インサイト）」を導き出すシステムであることを強調。
  • TEPSCO Knowledge Hub (テプスコ-ナレッジハブ)
    各部門の専門知識が交差し、つながる「中心地（ハブ）」としての役割を表現。

2. AIパートナー・協働を強調

  • TEPSCO Buddy (テプスコ・バディ)
    技術者の横に常に寄り添い、業務をサポートする「相棒」のような存在であることを親しみやすく表現。
  • TEPSCO Navigator (テプスコ・ナビゲーター)
    膨大な過去資料や業務の進め方において、正しい答えへと「導いてくれる」役割を強調。
  • TEPSCO AI-Assist (テプスコ・エーアイ・アシスト)
    機能そのものをストレートに表し、誰もが迷わず用途を理解できる堅実な名称。

3. 基盤・インフラとしての強固さを強調

  • TEPSCO Core (テプスコ・コア)
    これからの建設コンサルタント業務における、全く新しい「中核（コア）」となるシステムであることを表現。
  • TEPSCO Nexus (テプスコ・ネクサス)
    Nexus（結びつき、連鎖）の意味の通り、人とデータ、部門と部門を結びつけるプラットフォームであることを表現。

4. 洗練された略称・造語案

  • T-RAG (ティー・ラグ)
    TEPSCOの「T」とシステムの核となる技術「RAG」を掛け合わせた、プロフェッショナルで技術的な響きを持つ名称.
  • TepSync (テプシンク)
    TEPSCOとSynchronize(同期する)を組み合わせた造語。人とAI、データが常に同期して業務を進めるイメージ.
